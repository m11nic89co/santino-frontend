# Регламент фаз программы Rewrite

## Назначение

Документ фиксирует порядок выполнения фаз, критерии перехода между фазами и обязательные артефакты. Все агенты обязаны следовать этому регламенту.

## Границы изменений (write-boundaries)

- **Разрешено изменять только:**
  - `/Users/alexandrmelnicenco/Desktop/SC/rewrite/`
  - `/Users/alexandrmelnicenco/Desktop/SC/docs/rewrite/`
- **Запрещено:** любые изменения вне указанных директорий. Legacy-код — только чтение.

## Режим legacy (immutable)

- Legacy-кодовая база используется исключительно как **Read-Only Behavioral Specification**.
- Разрешено: читать файлы для извлечения требований и поведения.
- Запрещено: изменять, форматировать, копировать реализацию в rewrite, удалять legacy-файлы.

## Phase-gate (формат перехода между фазами)

1. Агент выполняет только задачи текущей фазы в разрешённых директориях.
2. Агент обновляет чеклист DoD в `REWRITE_PLAN.md` (галочки `[x]`).
3. Агент формирует отчёт по единому шаблону и передаёт на Architect Review.
4. Фаза считается закрытой **только** после статуса Architect: `APPROVED`.
5. Не пройден gate — фаза не закрыта; переход к следующей фазе запрещён.

## Нумерация и порядок фаз

Фазы выполняются строго по порядку: 0 → 1 → 2 → … → 12. Пропуск или параллельное выполнение недопустимы без явного решения Архитектора.

## Процесс эскалации блокеров

1. Если агент не может выполнить задачу фазы (неясные требования, конфликт правил, технический тупик), он **не вносит изменений** вне разрешённых зон и не нарушает границы.
2. Агент фиксирует статус **BLOCKED** в отчёте по фазе и описывает причину блокировки в разделе «Риски / блокеры / предложения».
3. Эскалация передаётся Архитектору (Principal Systems Architect). Решение Архитектора (уточнение требований, исключение, изменение регламента) фиксируется в журнале ADR при необходимости.
4. До снятия блокировки фаза не считается завершённой; переход к следующей фазе запрещён.
