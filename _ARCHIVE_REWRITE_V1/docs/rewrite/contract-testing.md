## 1. Pass/Fail критерии для P0 и P1

- **P0:** Контракты обязательны для релиза. Любой упавший P0-тест = gate failed; сборка/деплой не одобряются до исправления.
- **P1:** Контракты желательны; допускается временное исключение по решению архитектора с фиксацией в тикете и датой пересмотра.

Критерии по контрактам заданы в contracts-matrix (колонки «Выходы», «Измеримость»). В E2E:

- **Pass:** все утверждения в соответствующем spec-файле прошли (селекторы, атрибуты, видимость, данные).
- **Fail:** хотя бы одно утверждение не выполнено или таймаут (элемент не найден / не видим в заданное время).

## 2. Допуски для визуальных сравнений

- Визуальные тесты (snapshot / screenshot) выполняются только в рамках, явно добавленных в контрактный набор (если будут введены).
- **Допуск по пикселям:** различие ≤ 0.5% от площади кадра при неизменных шрифтах и разрешении — допустимо (шрифтовой рендер, антиалиасинг).
- **Исключаемые области:** таймеры, даты, анимированные части (тикер, счётчики) — маскировать или исключать из сравнения.
- **Baseline:** хранить в `e2e/snapshots/` (или аналог); обновление baseline только через явный скрипт и код-ревью (например, `npm run test:e2e -- --update-snapshots`).
- Политика: при визуальном регрессе — сначала анализ (ожидаемое изменение или баг); при ожидаемом — обновление baseline; при баге — исправление кода.

## 3. A11y и reduced-motion контракты

- **C-A11Y-01 (Skip Link & Focus):** наличие `.skip-link` с `href="#main-content"`; наличие landmark `#main-content` или `main`; фокус при Tab — измеримо в E2E.
- **C-A11Y-02 (Reduced Motion):** при `prefers-reduced-motion: reduce` у `html` должен быть класс `reduced-motion`; анимации (тикер, интро) приостановлены — проверка через `page.emulateMedia({ reducedMotion: 'reduce' })` и проверку класса/отсутствия анимации.
- **C-A11Y-03 (Section/Swiper ARIA & Live Regions):** секции с заголовками или `aria-labelledby`; активный слайд/навигация с корректным `aria-current="page"` и при необходимости live region — измеримо по DOM и атрибутам.

Ожидания по контрактам совпадают с разделом «Измеримость» в contracts-matrix.

## 4. Протокол обработки flaky-результатов

- **Определение:** тест считается flaky, если при тех же коммите/окружении он то проходит, то падает (без изменения кода теста или приложения).
- **Действия:**
  1. Зафиксировать в тикете: Contract ID, имя теста, последние логи/артефакты (trace, screenshot).
  2. В CI: retry упавшего теста 1 раз (настроено в `playwright.config.ts`: `retries: process.env.CI ? 1 : 0`).
  3. Если после retry тест прошёл — считать прогон зелёным, но оставить тикет на стабилизацию.
  4. Если тест стабильно падает в нескольких прогонах — не помечать как flaky; считать реальным падением контракта.
  5. Решение о временном отключении теста или пометке как flaky принимает ответственный (архитектор/владелец среза); отключённый тест должен иметь комментарий с тикетом и датой пересмотра.

## 5. Формат отчетности по контрактам

- **Таблица:** Contract ID | Приоритет (P0/P1) | Результат (Pass/Fail/Skipped) | Spec файл | Примечание.
- **Артефакты:** HTML-отчёт Playwright (`playwright-report/`), при падении — trace и screenshot в артефактах CI.
- **Итог:** блокирующим считается только полный проход всех P0; P1 допускают исключения по процессу выше.

Пример строки отчёта:

| Contract ID | Приоритет | Результат | Spec | Примечание |
|-------------|-----------|-----------|------|------------|
| C-APP-01    | P0        | Fail      | e2e/p0/contract-app.spec.ts | Loader not hidden — v2 not implemented |

После реализации фич ожидается переход в Pass без изменения критериев контракта.
