# REWRITE_PLAN.md

## Стратегический статус

- Принятое решение: **Полный Rewrite**.
- Legacy-кодовая база переводится в режим: **Read-Only Behavioral Specification**.
- Цель: построить новую кодовую базу без наследования технического долга и без смешивания со старой реализацией.

---

## 0. Непереговорные правила программы Rewrite

### 0.1 Физическая изоляция (обязательная)

- Новая кодовая база размещается **только** в:
  - `/Users/alexandrmelnicenco/Desktop/SC/rewrite/`
- Дополнительные документы программы — только в:
  - `/Users/alexandrmelnicenco/Desktop/SC/docs/rewrite/`
- Любые изменения вне этих директорий запрещены.

### 0.2 Режим legacy: read-only

- Разрешено: читать legacy для извлечения требований и поведения.
- Запрещено: изменять, форматировать, переносить, “слегка поправить”, удалять legacy-файлы.
- Запрещено копировать legacy-реализацию в rewrite (допустимо только функциональное переосмысление по контрактам).

### 0.3 Технологический стандарт v2

- Строгий TypeScript (strict mode, без послаблений).
- Vite.
- Строгий ESLint + Prettier.
- CSS Layers + token-based styling.
- Архитектурные запреты:
  - `window.*` в бизнес/feature-слоях.
  - Таймеры-костыли как способ синхронизации логики.
  - Inline-стили как основной механизм состояния UI.
  - Прямое обращение feature-слоев к vendor API без адаптеров.

### 0.4 Управление качеством

- Ни одна фаза не закрывается без прохождения собственного DoD.
- Каждая фаза завершается только после Architect Review и явного статуса: `APPROVED`.
- Формат отчета агента обязателен (см. раздел “Единый шаблон отчета”).

## 1. Целевая архитектура v2 (State-of-the-Art)

## 1.1 Behavioral Contract Suite
Контрактный слой, фиксирующий ожидаемое поведение продукта (как внешнюю спецификацию), независимо от legacy-реализации.

## 1.2 Runtime Kernel
Детерминированное ядро исполнения: lifecycle, state transitions, типизированные события, orchestration.

## 1.3 Vendor Adapters
Антикоррупционный слой (Ports & Adapters), изолирующий внешние библиотеки и их API.

## 1.4 Feature Slices
Вертикальные срезы функциональности с четкими границами ответственности и зависимостей.

## 1.5 Dependency Direction (обязательное направление зависимостей)
`feature slice -> application contracts -> runtime kernel ports -> vendor adapters`
  
Обратные зависимости и обходы запрещены.
