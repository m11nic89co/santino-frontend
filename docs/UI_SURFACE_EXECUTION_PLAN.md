# План исполнения: поверхности UI (Hero / Контракт / Контакты)

Документ фиксирует **пошаговое исполнение** рекомендаций UI-аудита (разделение токенов, глубина контракта, оптический центр hero, контакты, a11y/perf) с **чекпоинтами в файле** и **матрицей тестов** для контроля полноты.

---

## 1. Цели и границы

| Цель | Критерий |
|------|----------|
| Иерархия секций | Три осмысленно различимые поверхности (роли), без «одной панели на всё» |
| Геометрия | Нет лишних пустот из‑за общего `min-height`; минимумы только по смыслу (сцена mold и т.п.) |
| Контракт | Читаемость SVG/HUD на фоне; восстановлена визуальная глубина (не обязательно 1:1 старый radial) |
| Hero | Оптический баланс с учётом шапки, тикера, доверительной строки |
| Контакты | Согласованное выравнивание десктоп/мобайл + читаемость длинного текста |
| Регрессия | Сборка, смоук, визуальные эталоны по пресетам |

**Вне скоупа этого плана (если не согласовано отдельно):** пуш на GitHub Pages, смена копирайта, новые секции.

## 3. Фазы исполнения (порядок)

### Фаза 0 — Подготовка (чекпоинт **C0**)

1. Зафиксировать текущее состояние: коммит или заметка `git rev-parse HEAD`.
2. Снять **baseline** визуальные эталоны (см. §5.1).
3. Убедиться, что `npm run build` и `npm run test:smoke` проходят на baseline (зафиксировать, если падает — завести задачу).

**Выход:** заполнен C0, файлы baseline в `ScreenShots/auto-sections/` (или путь из §5.1).

---

### Фаза 1 — Токены поверхностей (чекпоинт **C1**)

1. Ввести минимум **три** набора переменных (пример имён): `--ui-surface-hero-*`, `--ui-surface-tech-*`, `--ui-surface-info-*` (фон, blur, border, при необходимости радиус/паддинги).
2. Подключить: Hero → `hero`, стенд mold-reveal → `tech`, карточка контактов → `info`.
3. Задокументировать в комментарии у `:root` назначение каждой группы (1–2 строки).

**Выход:** визуально секции различимы; код не дублирует магические rgba в трёх местах без смысла.

---

### Фаза 2 — Min-height по контексту (чекпоинт **C2**)

1. Убрать или снизить **общий** `--ui-surface-min-h` с карточек, где он создаёт пустоту (контакты, hero-текст).
2. Оставить/уточнить минимум только для композиции **сцены контракта** (`.kinetic-blueprint-stand` / `.ghost-layer` / `reveal-stage` — по факту вёрстки).
3. При необходимости: `max-height` + скролл для длинного текста контактов на узких высотах viewport.

**Выход:** нет «вытянутых пустых стёкол» на типичном десктопе 1080p.

---

### Фаза 3 — Контракт: глубина (чекпоинт **C3**)

1. Добавить лёгкий **внутренний скрим/градиент/виньетку** под сцену (или под всю площадь стенда), не ломая glass.
2. Обновить устаревшие комментарии в `mold-reveal.css` (ссылки на «градиентный блок» → актуальная формулировка).
3. Проверить позиционирование CTA относительно новой геометрии (десктоп/мобайл).

**Выход:** HUD и SVG читаются лучше, чем на «голом» 2% white glass.

---

### Фаза 4 — Hero: оптический центр (чекпоинт **C4**)

1. Задать безопасную зону: центрирование в `calc(100dvh − header − нижняя_полоса)` или смещение на N px после сравнения «до/после».
2. Не ломать абсолютные `.hero-subtle` и `.ticker-wrap`; согласовать с `padding-bottom` слайда.
3. Зафиксировать выбранную формулу коротким комментарием в CSS.

**Выход:** визуально центр блока не «провисает» относительно сетки экрана на 1920×1080.

---

### Фаза 5 — Контакты (чекпоинт **C5**)

1. Явно выбрать паттерн десктопа: центрированный заголовок + левая колонка реквизитов, или единая сетка с одним краем.
2. Подстроить **плотность** `--ui-surface-info-*` под длинный текст (чуть выше фон / меньше «шума» карты через контраст).
3. Сверить с мобильным поведением (`#contacts`), чтобы не было ощущения «два разных экрана».

**Выход:** читаемость абзацев и ссылок без напряжения; согласованность с остальным сайтом.

---

### Фаза 6 — A11y и производительность (чекпоинт **C6**)

1. Контраст основного текста в карточках (ручная проверка или инструмент) — целевой уровень **не ниже** существующей политики проекта (`docs/architecture/a11y-motion-compliance.md` при необходимости).
2. При просадках FPS: сократить число тяжёлых `backdrop-filter` или ограничить область blur (без полной потери концепции).

**Выход:** чеклист §5.3 отмечен.

---

### Фаза 7 — Финальная регрессия (чекпоинт **C7**)

1. Повторить полный набор тестов §5 (все обязательные).
2. Обновить таблицу чекпоинтов C0–C7 статусом `✅`.
3. Краткая запись в «Примечание» к C7: версия коммита, кто принял визуально (по процессу команды).

## 5. Команды и артефакты

### 5.1 Визуальные эталоны (Playwright-скрипт)

Базовый URL зависит от окружения (Vite dev/preview). Примеры:

```bash
# Превью прод-сборки (base path из vite.config)
npx vite preview --host 127.0.0.1 --port 4173
node scripts/capture-live-section.mjs --all --url http://127.0.0.1:4173/santino-frontend/ --label baseline-ui-plan
```

После правок:

```bash
node scripts/capture-live-section.mjs --all --url http://127.0.0.1:4173/santino-frontend/ --label phase-N-after
```

Дополнительный размер экрана:

```bash
node scripts/capture-live-section.mjs --preset contract --viewport 1366x768 --url … --label laptop-contract
```

Артефакты: `ScreenShots/auto-sections/*.png` и одноимённые `.json` (клип, viewport).

### 5.2 Автоматические

```bash
npm run build
npm run test:smoke
npm run lint
```

### 5.3 Чеклист A11y/Perf (ручной, для C6)

- [ ] Текст контактов читается без «мыльного» фона хуже, чем до изменений (или сознательно улучшен контраст).
- [ ] Фокус с клавиатуры доходит до интерактивов в трёх секциях.
- [ ] `prefers-reduced-motion`: нет новых обязательных анимаций, ломающих layout.
- [ ] Переключение слайдов 0→2→3→0 без заметных лагов на целевой машине.
